## 1. 改进目标

当前项目是本地 Pygame 单机游戏，没有后端、数据库、网络通信和账号系统。目标是升级为支持公网联机的双人模式，并增加名称登录、排行榜、自建房功能。

目标功能：

- 玩家可以输入名称登录。
- 玩家可以创建房间。
- 玩家可以通过房间号加入房间。
- 每个房间最多 2 人。
- 两名玩家可以不在同一局域网，只要能访问公网服务器即可联机。
- 游戏结束后保存分数。
- 客户端可以查看排行榜。

结论：

```text
当前项目不能只靠静态资源部署实现这些功能。
需要新增后端服务、数据库和实时通信。
```

## 2. 推荐整体架构

推荐第一版使用：

```text
Flask + Flask-SocketIO + MySQL
```

原因：

- 当前项目已经是 Python，接入成本低。
- Flask 适合做登录、排行榜、房间列表等 HTTP 接口。
- Flask-SocketIO 适合做双人实时同步。
- MySQL 作为第一版正式数据存储，适合公网部署和后续扩展。
- 本地开发需要先安装 MySQL，并配置数据库连接环境变量。

整体结构：

```text
玩家A Pygame客户端
        │
        ├── HTTP：登录、排行榜、创建房间、加入房间
        └── Socket.IO：实时发送输入、接收游戏状态
                    │
                    ▼
公网后端服务器 Flask + Flask-SocketIO
                    │
                    ├── 用户/名称记录
                    ├── 房间管理
                    ├── 游戏状态计算
                    ├── 排行榜保存
                    └── MySQL 数据库
                    ▲
        ┌── HTTP / Socket.IO
        │
玩家B Pygame客户端
```

### 2.1 需要先统一的设计决策

下面这些点会影响后端、客户端和数据库结构。为了避免开发时反复改底层，第一版建议先按表里的默认值执行。

| 问题 | 第一版建议 | 原因 |
| --- | --- | --- |
| 登录方式 | 只输入名称登录，不做密码 | 降低第一版复杂度，先跑通房间和联机 |
| 房间人数 | 固定 2 人 | 和当前目标一致，房间逻辑最简单 |
| 开始条件 | 两人都进入房间并点击 ready | 避免房主误开局 |
| 游戏分数 | 两人共享一个房间总分 | 排行榜和结算简单 |
| 生命规则 | 两人各自血量，两人都死亡才结束 | 双人合作体验更好 |
| 道具归属 | 谁捡到谁生效 | 逻辑清晰，改动较小 |
| 升级火力 | 玩家个人升级 | 避免一个人捡道具导致两人状态混乱 |
| 追踪/暴走 | 玩家个人计时 | 和个人升级一致 |
| Boss 目标 | 服务端按规则选择目标，优先距离近或随机二选一 | 不能由客户端各自判断 |
| 敌机生成 | 只由服务端生成 | 保证双方一致 |
| 道具/宝石掉落 | 只由服务端随机 | 保证双方一致，分数可信 |
| 碰撞检测 | 只由服务端计算 | 防止客户端结果不一致 |
| 排行榜依据 | 游戏结束时服务端写入房间总分 | 不能相信客户端提交分数 |
| 状态同步 | 服务端广播完整状态快照 | 第一版最稳，方便调试 |
| 客户端预测 | 第一版不做 | 先保证正确，再优化手感 |
| 断线处理 | 第一版断线即暂停房间，超时销毁 | 实现简单，避免房间挂死 |
| 房间列表 | 第一版可选，优先房间号加入 | 减少 UI 和接口工作量 |

第一版需要明确的边界：

```text
客户端不再拥有最终游戏逻辑决定权。
客户端可以播放动画和音效，但不能决定敌机、掉落、碰撞、分数和游戏结束。
```

### 2.2 昵称登录规则 [已完成]

第一版登录只使用昵称，不做账号密码、不做注册页、不做短信/邮箱验证。

当前实现状态：

```text
状态：[已完成]
后端文件：backend/app.py、backend/auth.py
接口：POST /api/login
验证：python -m unittest test_login.py
测试结果：3 个测试通过
```

登录流程：

```text
1. 客户端启动后显示昵称输入框。
2. 玩家输入昵称并点击进入。
3. 客户端请求 POST /api/login。
4. 后端校验昵称格式。
5. 如果昵称不存在，创建玩家记录。
6. 如果昵称已存在，复用该玩家记录并更新 last_login_at。
7. 后端返回 player_id、name、token。
8. 客户端后续 HTTP 和 Socket.IO 请求都携带 player_id 和 token。
```

昵称规则：

```text
长度：2-12 个字符
允许：中文、英文、数字、下划线
不允许：空格、表情、特殊符号
大小写：英文按原样保存，第一版不做大小写归一
重复昵称：允许再次登录同一个昵称，视为同一个玩家
```

重复登录处理：

```text
同一个昵称再次登录时，不新建玩家。
后端返回已有 player_id，并刷新 last_login_at。
如果同一昵称已经在房间或游戏中，第一版允许新连接顶掉旧连接。
```

token 规则：

```text
第一版使用服务端生成的随机 token。
token 保存在 players 表中。
客户端内存保存即可，第一版不要求本地持久化。
服务端收到需要身份的请求时，用 player_id + token 校验。
```

第一版不做：

```text
密码
注册
找回账号
游客转正
多设备同时在线
复杂会话过期策略
JWT
OAuth
```

后续如果要正式账号系统，再升级为账号密码或第三方登录。

## 3. 为什么需要后端

名称登录、排行榜、自建房和公网双人联机都需要一个双方都能访问的中间服务。

后端负责：

- 保存玩家名称。
- 生成房间号。
- 维护房间人数。
- 判断谁在房间里。
- 保存排行榜。
- 同步两名玩家的操作。
- 统一生成敌机、Boss、道具。
- 统一计算碰撞、死亡、分数和游戏结束。

如果没有后端，让两个客户端各自运行游戏逻辑，会出现：

- 敌机随机生成不一致。
- 子弹位置不一致。
- 碰撞结果不一致。
- 分数不同步。
- 房间状态无法统一。
- 排行榜无法可信保存。

因此联网版必须把“真正的游戏状态”放到服务器或至少由服务器统一校验。

## 4. HTTP 与 WebSocket 的分工

### 4.1 HTTP 适合处理的功能

HTTP 是一次请求一次响应，适合低频业务：

```text
登录
创建房间
加入房间
退出房间
查询房间列表
查询排行榜
提交最终分数
```

建议接口：

```text
POST /api/login
GET  /api/rankings
POST /api/rooms
GET  /api/rooms
POST /api/rooms/<room_id>/join
POST /api/rooms/<room_id>/ready
POST /api/rooms/<room_id>/leave
```

### 4.2 Socket.IO / WebSocket 适合处理的功能

实时游戏同步需要持续通信，适合用 Socket.IO / WebSocket：

```text
进入房间
准备状态
玩家输入
服务端广播游戏状态
游戏开始
游戏结束
断线通知
```

建议事件：

```text
client -> server:
- join_room
- player_ready
- input
- leave_room

server -> client:
- room_update
- game_start
- game_state
- game_over
- player_disconnected
```

## 5. 服务端权威状态设计

联网游戏最重要的原则：

```text
服务器是权威状态源，客户端只负责输入和显示。
```

客户端负责：

- 读取键盘输入。
- 把输入发给服务器。
- 接收服务器发来的游戏状态。
- 根据状态绘制玩家、敌机、子弹、Boss、道具和 UI。

服务器负责：

- 每个房间维护一份游戏状态。
- 根据玩家输入更新玩家位置。
- 自动生成敌机、Boss、道具。
- 自动生成玩家子弹和 Boss 子弹。
- 计算碰撞。
- 计算血量和分数。
- 判断游戏结束。
- 周期性广播状态给两个客户端。

推荐服务端 Tick：

```text
30 tick/s 起步
后续优化可提升到 60 tick/s
```

Pygame 客户端本地仍可 60 FPS 绘制，但网络状态可以 20-30 次每秒同步。

### 5.1 随机生成如何保持统一

双人联机里，敌机、Boss、道具、宝石掉落不能由两个客户端分别随机生成。否则即使用同一份代码，双方也会因为帧率、网络延迟、随机调用次数不同，很快出现画面不一致。

第一版采用最稳的方案：

```text
所有随机事件只在服务端发生。
客户端不生成敌机、不生成道具、不判定掉落。
客户端只接收服务端广播的结果并绘制。
```

服务端负责统一决定：

- 哪一帧生成敌机。
- 生成几只敌机。
- 敌机类型。
- 敌机初始坐标。
- 敌机速度。
- 是否触发 Boss。
- Boss 出现时机和血量。
- Boss 技能触发。
- 敌机死亡后掉落哪些宝石。
- 是否掉落功能道具。
- 道具类型。

服务端广播给客户端的是已经确定的结果，例如：

```json
{
  "type": "spawn_enemy",
  "tick": 320,
  "enemy": {
    "id": "e_102",
    "enemy_type": 2,
    "x": 180,
    "y": -30,
    "speed": 1.6,
    "hp": 3
  }
}
```

客户端收到后只创建一个“显示对象”，不再重新随机。

后续如果要节省网络流量，可以升级为“服务端发随机种子 + tick + 事件序号”的确定性同步方案。但第一版不建议这样做，因为当前游戏逻辑还没有完全确定性，直接用服务端广播完整状态更稳。

### 5.2 第一版同步颗粒度

第一版建议使用“服务端广播完整状态快照”，不要一开始就做复杂的帧同步。

服务端每 tick 或每隔固定 tick 广播：

```text
players
player_bullets
boss_bullets
enemies
items
boss
score
game_state
```

客户端本地只做：

```text
1. 读取键盘输入。
2. 发送输入到服务端。
3. 保存最新收到的 state。
4. 根据最新 state 绘制画面。
```

这种方式的优点：

- 双方画面一定以服务端为准。
- 随机事件不会分叉。
- 排行榜分数可信。
- 调试简单。

缺点：

- 状态包比帧同步更大。
- 延迟高时玩家会感觉操作有一点滞后。
- 后续需要插值或客户端预测优化手感。

第一版可以接受这些缺点，先保证功能正确。

### 8.1 players 表

```text
id              BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT
name            VARCHAR(12) UNIQUE NOT NULL
token           CHAR(64) UNIQUE NOT NULL
is_online       TINYINT(1) DEFAULT 0
current_room_id VARCHAR(16)
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
last_login_at   TIMESTAMP NULL
```

第一版只做昵称登录，不做密码。`token` 用于客户端后续请求身份校验，`current_room_id` 用于判断玩家当前是否已经在房间内。

如果需要正式账号系统，再增加：

```text
password_hash TEXT
token_version INTEGER
```

### 8.2 scores 表

```text
id          BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT
player_id   BIGINT UNSIGNED NOT NULL
score       INT UNSIGNED NOT NULL
mode        VARCHAR(32) NOT NULL
room_id     VARCHAR(16)
created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
```

### 8.3 rooms_history 表

```text
id             BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT
```

## 11. 双人玩法改造点

当前单机项目只有一个 `hero`。

联网双人需要改成：

```text
players = {
  player_id_1: hero_state_1,
  player_id_2: hero_state_2
}
```

需要改造：

- 英雄状态从单对象变成双对象。
- 子弹需要记录所属玩家。
- 道具拾取需要记录拾取玩家。
- 碰撞需要分别判断两个玩家。
- 失败条件需要明确：
  - 两个玩家都死亡才结束；
  - 或任意玩家死亡就结束；
  - 或共享生命值。
- 分数需要明确：
  - 两人共享分数；
  - 或分别计分；
  - 排行榜按房间总分还是个人分。

建议第一版：

```text
两人共享分数。
两人各自有血量。
两人都死亡才 game_over。
排行榜记录房间总分。
```

## 15. 技术风险

### 15.1 当前游戏逻辑耦合较重

`test.py` 中混合了主循环、状态机、绘制、碰撞、音频和 UI。直接接网络会很难维护。

建议先把游戏状态和渲染逻辑分开：

```text
GameState 数据：可序列化，可通过网络传输
Render 渲染：只根据状态画图
Input 输入：只收集按键并发送
```

### 15.2 网络延迟

公网 WebSocket 会有延迟。第一版可以接受轻微延迟，后续再做客户端预测和平滑插值。

第一版建议：

- 服务端 30 tick/s。
- 客户端 60 FPS。
- 客户端直接绘制最近一次收到的状态。
- 后续再加入插值。

### 15.3 数据可信问题

排行榜不能完全相信客户端提交的分数。正式版应该由服务端计算分数并写入数据库。

### 15.4 服务器性能

Pygame 的精灵和碰撞逻辑主要为本地渲染设计，不一定适合直接搬到服务端。服务端最好用轻量数据结构表示对象：

```text
id, type, x, y, vx, vy, hp, active
```

不要在服务端依赖图片、音效和 Pygame Surface。

### 15.5 随机事件不同步

这是双人实时模式的核心难点之一。

当前单机代码里敌机生成、敌机类型、敌机位置、掉落道具、宝石掉落都依赖 `random`。如果两个客户端分别运行这些逻辑，即使初始代码一样，也会因为下面原因分叉：

- 两边帧率不同。
- 网络延迟不同。
- 一方暂停或卡顿。
- 两边调用 `random` 的次数不同。
- 一边先发生碰撞，另一边后发生碰撞。

解决原则：

```text
客户端不允许自行随机生成会影响玩法的对象。
所有影响玩法的随机事件由服务端统一决定。
```

第一版具体做法：

- 服务端生成敌机并分配唯一 id。
- 服务端生成道具和宝石并分配唯一 id。
- 服务端计算 Boss 技能和目标选择。
- 服务端计算碰撞和分数。
- 客户端收到服务端状态后，只负责绘制这些对象。

不要第一版就做客户端同步随机种子。种子同步要求游戏逻辑完全确定性，而当前项目还没有达到这个条件。
